Nexys3 – boards and vhdl basics

How to program?

$ djtgcfg prog -f top.bit -d Nexys3 -i 0

Learn the VHDL for, if, generate, syntax:

btn_debounce_for : for i in 0 to 4 generate
   UBTN : btn_debounce port map(clkin,gpio_btn(i),btn_internal(i));
end generate btn_debounce_for;

for – generate loop also supports process, and moreover, it supports nested if – generate statement,

led_for : for i in 0 to 4 generate
    p_led_latch : process(clkin) is
    begin
        if rising_edge(clkin) then
            if btn_internal(i)='1' then
                led_internal(i) <= not led_internal(i);
            end if;
        end if;
    end process p_led_latch;
    gpio_led(i) <= led_internal(i);
end generate led_for;

We may also infer combinational logic use with – select clause,

with seg select
    bcd <= "10001110" when "1111";
           "11111111" when others;

Note: we should place clk pins on GCLK sites, where C-1 (H3), B-9 (K3), B-10 (K5). We wrap B9 and B10 to XCK and PCK. And we add insulate via Tin foil (锡铂纸).

By comparing PX4Flow and the mt9v032 taobao board, we found some common connections, OE (active HIGH) tied to VCC, RSVD tied to GND. BYPASS_CLKIN_P and SER_DATAIN_P are tied to GND, while BYPASS_CLKIN_N and SER_DATAIN_N are tied to VCC with 1k Ohm pull-up resistor.

However, for mt9v032-dev, the STANDBY is tied to GND, and the RESET_N is tied to VCC with 10k Ohm pull-up resistor. For PX4Flow, these two pins are controlled by MCU. For mt9v032-dev, SCTRL_ADDR0 and SCTRL_ADDR1 are tied to GND, while for PX4Flow, these pins are tied to VCC.

These two boards works in master mode, with steroscopy mode disabled. SCK and SDA should be pulled-up with 1.5K Ohm resistors when no other I2C peripheral is connected.

NIR – boot mt9v032 via nexys3

In principles, the mt9v032 works output wide-VGA image at 60fps by default.

By injecting DCMI_SYS_CLK with 25MHz clock input, and trigger DCMI_EXP with a rising-edge sensitive pulse (1000 cycles duty period, in snapshot or slave mode; in master mode, DCMI_EXP may tie to GND). We may monitor the DCMI_DATA bus, DCMI_HSYNC and DCMI_VSYNC signals, as well as DCMI_LED pins.

DCMI_VSYNC is HIGH valid during the whole imaging process (a frame), DCMI_HSYNC is periodorically HIGH valid. The image format is 752x480. However, currently DCMI_CLK could not register exactly 752 datum each HSYNC pules. (need debug this). Strange, DCMI_CLK may be 26.6MHz PLL locked even if we inject a 25MHz DCMI_SYS_CLK.

NIR – the wrapper of i2c_io_16bit.vhd

With ADDR0 and ADDR1 pins pulled low on mt9v032-dev, the address is 0x48, with the last bit denotes write (’0’) or read (’1’). Therefore, the mt9v032 board has 0x90 for write, 0x91 for read. We build dedicated module i2c_io_16bit.vhd to handle the Read/Write process to mt9v032.

0x90 (0x91) REG_ADDRESS WB1 (RB0) WB2 (RB0)

for example, 0x90070188 write 0x0188 to 0x07 in mt9v032, 0x9107XXXX read DBYTE from 0x07 in mt9v032.

As a simple rule-of-the-thumb, we may put 0x91000000 on data bus, trigger data_stream_in_stb, and read out the content of 0x00 on mt9v032 by acknowledge data_stream_out_stb, and finally put the register (which is the CHIPID, 0x1324) on BCD segment display.

Another simple example is command 0x900400FF, which changes the WINDOW_WIDTH to 256, this can also be verified via Chipscope.

NIR – config mt9v032 (copied from px4flow)

After testing the R/W functionality of i2c_io_16bit.vhd, we next store all the command and registers in the mt9v032.coe, and loaded in a rom to config this camera module. The top module, which is called mt9v032.vhd, embedded the mt9v032.coe, sequentially read the content out, and config the device via I2C protocol.

We port the source code mt9v032.c in PX4Flow to NIR project, (Context A, context, is like a fast stack-switch machanism, context B in PX4Flow is used for videos streaming only, while context A is for practical optical flow calculation).
